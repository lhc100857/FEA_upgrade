# 主帅上下文生命周期管理协议 v2（通用版 · 平台无关）

> 本文档独立于任何特定仓库结构。它描述的是一套**概念协议**，适用于任何采用
> 「一个主编排者 + 多个子代理」模式的 AI 协作体系（蜂群 / 军团 / Swarm / Legion / 
> Multi-Agent Orchestration / 类似架构）。

---

## 问题陈述

在多代理协作体系中，主编排者（统帅 / Orchestrator / Commander）因以下原因导致**上下文窗口膨胀**，进而引发幻觉、部分遗忘与形状漂变（即已冻结的决策被后续轮次静默覆盖）：

1. **全量重述**：每轮将所有冻结事实、验收状态、待决项全部重新写入对话
2. **冻结条只增不减**：已失效的决策永远留在上下文中
3. **子代理回流过长**：子代理返回完整源码、日志等大量文本
4. **结构性开销**：治理规则文件本身占用 ~7-15% 上下文窗口
5. **Lost in the Middle**：中间位置的信息被模型低权重处理

## 核心方案：三层上下文模型

将主编排者的记忆从「全部在对话里」分离为三层：

| 层 | 名称 | 预算建议 | 内容 | 刷新方式 |
|---|---|---|---|---|
| **L1** | 工作记忆 (Working) | ≤30k token（基础）/ ≤45k token（复杂编排） | 当轮增量、用户最近 2 轮原文、活跃验收清单、子代理回流 | 每轮扫描；超预算时驱逐至 L2 |
| **L2** | 情节记忆 (Episodic) | ≤8k token | 全链冻结条状态表、阶段门裁决、条件追踪、归档指针 | 阶段切换 / 门禁通过时压缩 |
| **L3** | 语义记忆 (Semantic) | 无上限（磁盘/外存） | 跨会话决策、子代理完整产出、原始工具输出、归档冻结条全文 | 追加写入；新会话按需加载 |

**预算守恒参考**：L1 + L2 + 规则税 + 系统开销 ≤ 有效上下文窗口的 50-60%。

> 具体数值取决于你的模型与平台（Claude ~200k 但质量拐点更早，GPT-4 ~128k，Cursor 实测 ~120k 质量拐点）。按各自平台调整。

## 组件一：链状态锚点文件（CHAIN-STATE）

每条任务链维护一个**磁盘文件**作为 L2 层的物理锚点。主编排者不再靠对话记忆冻结条，而是读写这个文件。

**前 20 行为「精简下限区」**——主编排者仅 Read 此区即可回答：目标是什么、当前验收状态、关键 P0、身份三字段。

### Schema（七区）

```markdown
# CHAIN-STATE — {链ID}
updated: {时间戳}  round: {N}  mode: {基础|复杂编排}

## §A 精简下限区
- **目标句**: {一句话}
- **验收裁决**: {PASS / COND_PASS / FAIL + 编号}
- **active P0**: {列表}
- **身份三字段**: {执行者} | {人设} | {用途}

## §B 冻结条（四态生命周期）
| ID | 摘要 | 等级 | 状态 | 来源轮次 | 废止指针 |

## §C 阶段门裁决区（复杂编排专用）
| Gate | 裁决 | 条件ID | 理由 | 轮次 |

## §D 条件追踪区（≤8条open硬上限）
| COND_ID | 条件 | 状态 | 关联 | 截止 |

## §E 编排结构指针区（复杂编排专用）
- pinned_level: full|skeleton|id-only
- 任务书路径 / 架构路径 / 调度路径

## §F 子任务归档区（复杂编排专用）
| 子任务 | 角色 | 摘要 | 落盘路径 | →§B指针 |

## §G 最近delta（保留最近2轮）
```

**落位建议**：放在你的治理体系的「落盘 / 持久化」目录下，按链 ID 分子目录。模板见随附 `CHAIN-STATE-TEMPLATE.md`。

## 组件二：增量 delta 汇流（替代全量重述）

每轮主编排者的「任务状态续聊」改为**增量格式**：

```
**Δ ROUND={N}**
- [+] 新增: {内容}
- [~] 变更: {旧→新}
- [=] 验收进展: {编号 状态变更}
- [!] 待决/告警: {内容}
- [!CONFLICT] 矛盾: {描述 vs 已有}
- [→] 下轮指令: {内容}
```

### 不可省段（每轮必出现）
- 目标句首行回显
- active P0 状态变更
- 验收裁决变更（若有）

### 可省段
- 未变化的冻结条正文
- 已通过的验收条目详情
- 已归档/已替代的条目

**与 CHAIN-STATE 联动**：delta 同步 patch 到 CHAIN-STATE §G 区。

---

## 组件三：冻结条四态生命周期

| 状态 | 含义 | 转换条件 |
|------|------|----------|
| **active** | 当前有效 | 新建或恢复 |
| **superseded** | 被新条替代 | 主编排者显式声明 + 指向替代条 ID |
| **archived** | 不再相关 | 压缩时仅留 1 行墓碑；墓碑 >20 行时清理到最近 5 行 |
| **invalidated** | 被破坏性变更推翻 | 关联阶段门裁决同时标 invalidated，须重验 |

---

## 组件四：周期快照与压缩

**触发条件（三选一先到先触发）**：
1. 每 3 轮编排回合
2. 主对话估算 ≥50k token
3. 事件：验收失败重跑、链切换、用户明示存档

**压缩后验证**：主编排者须 Read CHAIN-STATE §A 区，确认 active P0 数量未变且目标句未丢。

---

## 组件五：子代理回流格式约束

嵌入每个子代理任务描述的尾部：

```
=== 回流格式约束 ===
你的最终回复须≤800字，结构如下：
1. 【结论】≤3条，每条≤50字，标可验证等级
2. 【交付物指针】落盘路径列表
3. 【下游依赖】须传递给哪个子任务的哪些字段
4. 【风险/待决】≤2条
禁止：返回完整源码、raw日志、重复任务书全文
```

**归档流程**：
1. 子代理返回（≤800 字）
2. 主编排者提取结论 → 写入 CHAIN-STATE §B
3. 摘要 → 写入 §F
4. 完整产出已在子代理内落盘（L3）
5. 门禁检查 → 通过则下一子任务读 §F 指针

## 组件六：常驻内容分级摘要

适用于有「任务书 / 架构文档 / 调度图」等需常驻上下文的编排体系。

| 级别 | 形态 | ~token | 适用时机 |
|------|------|--------|----------|
| **full** | 完整文本 | 2000-4000 | 首轮冻结、架构变更、破坏性审查 |
| **skeleton** | 角色名+依赖箭头+关键条ID+非目标一句 | 500-800 | 常态执行 |
| **id-only** | 版本号+关键条计数 | ~50 | 上下文紧张时 |

**降级触发**：L1 > 预算 80% → full→skeleton；> 95% → skeleton→id-only。
**恢复**：用户提及架构/任务书 → 从 L3 加载回全文。
**校验**：降级时输出 `[PINNED-DOWNGRADE]` 并验证关键条计数。

## 组件七：会话切割与接力

| 信号 | 阈值建议 | 动作 |
|------|----------|------|
| 轮次 | ≥15（复杂）/ ≥20（基础） | 建议切割 |
| token | ≥50k 估算 | 强烈建议 |
| 死循环 | 同一 P0 连续 3 轮失败 | 强制切割 |

### 切割时写 HANDOFF 文件
内容 = 全量 active 冻结条 + P0 + 阶段门进度 + 最近 2 轮 delta + 下轮指令。

### 新会话启动优先级
HANDOFF（若存在）> CHAIN-STATE > 从头重建。

## 组件八：平台压缩联动（用户确认制 · wave-end 建议触发）

多数 AI 平台提供对话压缩功能（内置摘要命令、开新对话等）。
本组件将平台压缩与 CHAIN-STATE 协议联动，确保压缩后不丢关键状态。

**核心原则：压缩决策权归用户**。主编排者只负责写盘和建议，**不得**自行触发压缩或假设用户会执行。

### 四步安全流程

```
1. 【写盘】主编排者更新 CHAIN-STATE 全部相关区段
2. 【询问】主编排者向用户输出压缩建议，等待用户确认
3. 【压缩】用户确认后，执行平台压缩命令（或开新对话）
4. 【校验】主编排者 Read CHAIN-STATE §A，确认 4 项未丢：
   ✓ 目标句  ✓ active P0  ✓ 验收裁决  ✓ 身份三字段
   → 不匹配 → Read §B+§G 重建 L1
```

### 询问话术模板

主编排者在 wave-end 或达到压缩阈值时，向用户输出：

```
本轮 {wave/回合} 已完成，关键状态已持久化到 CHAIN-STATE。
当前上下文预估 {约XX}k token，建议执行上下文压缩以释放空间。

你可以：
1. 执行压缩命令（如平台内置的摘要/summarize 功能）— 我会在压缩后校验状态完整性
2. 暂不压缩，继续当前会话
3. 切割会话（我会先写 HANDOFF 文件）

请选择，或直接继续提问。
```

### 触发时机分层

| 场景 | 主编排者动作 | 理由 |
|------|-------------|------|
| **复杂编排 wave 结束（阶段门已裁决）** | **写盘 + 询问用户是否压缩**（首选时机） | 所有子任务已归档到 §F、阶段门已入 §C、COND 已更新 §D；下一 wave 需要新鲜 L1 |
| 基础编排每 3 轮 / token ≥50k | 写盘 + 询问用户是否压缩 | 与周期快照对齐 |
| 同一 P0 连续 3 轮失败 | **建议会话切割**（非压缩） | 深度失忆风险，需完整重建 |
| 编排链切换 | **建议会话切割** | 跨链污染风险 |
| 主编排者开始重复/车轱辘话 | 提示用户：先试压缩 | 轻度膨胀先压缩，无效再切割 |
| 平台自动触发压缩（非用户主动） | **立即** Read §A 校验 | 无法阻止时做好安全网 |

### 复杂编排 wave-end 专属检查清单

wave 的阶段门裁决完成后、**询问用户前**，主编排者须完成写盘：

1. **§C**：本 wave 的 Gate 裁决已写入（含 COND_ID 列表与理由）
2. **§D**：所有新产生/已消解的 COND 已更新状态
3. **§F**：本 wave 所有子任务的归档摘要已写入（含落盘路径和 →§B 指针）
4. **§B**：本 wave 产生的 active 冻结条已写入
5. **§A**：验收裁决和 active P0 已更新到当前状态
6. **§E**：pinned_level 已评估（若 L1 紧张，提前降级到 skeleton）

**全部写盘完成 → 询问用户 → 用户确认压缩 → 主编排者 Read §A 校验 → 开始下一 wave**

用户选择「暂不压缩」时，主编排者正常继续，**不得催促或重复建议**（下次触发条件满足时可再次建议）。

### 压缩后恢复（校验失败时）

```
§A 不匹配（目标句/P0/裁决/三字段任一丢失）
  → Read §B 恢复全量 active 冻结条
  → Read §G 恢复最近 2 轮 delta
  → Read §C+§D 恢复阶段门与 COND（复杂编排）
  → 在下一条回复中声明「已从 CHAIN-STATE 恢复」
```

### 压缩 vs 会话切割

| 情形 | 推荐 | 特点 |
|------|------|------|
| 上下文膨胀但进展正常 | 平台压缩 | 保留会话、成本低、轻量 |
| 深度失忆 / 跨链 / 连续失败 | 会话切割 | 写 HANDOFF、开新对话、重量级重建 |

两者互补，不互斥。

### 已知平台风险

- 部分平台会在非用户主动时**自动**触发上下文压缩（如上下文使用 ~60% 时）
- 复杂多步任务中压缩后失忆风险更高
- 压缩后工具状态（终端会话、临时变量等）可能丢失

**CHAIN-STATE 是所有风险的安全网**——即使平台自动压缩，只要磁盘文件在，状态可恢复。遇到非用户主动的自动压缩时，主编排者应**立即** Read §A 校验。

### 各平台等价操作参考

| 平台类型 | 压缩方式 | 备注 |
|----------|----------|------|
| 支持原生压缩命令的 IDE Agent | 平台内置命令 | 训练过的自摘要，效率最高 |
| 对话式 AI（无文件系统） | 开新对话 + 粘贴 §A | 手动接力，用 Artifact 或 code block 模拟 |
| API / 自建 Agent | 调用摘要模型 + 写 CHAIN-STATE | 可编程控制保留内容，最灵活 |
| 项目管理式 AI 平台 | 新 Thread + 附加 CHAIN-STATE 文件 | 按平台 Thread 机制操作 |

## 组件九：条件追踪硬上限

- open 条件 ≤8 条
- 第 9 条到来时：最旧 open → expired + 风险登记
- expired 条目不计入 8 条配额

---

## 组件十：上下文污染防御

- 冻结条须标可验证等级（可验证 / 待核实 / 观点）
- 待核实/观点不得在后续轮次被引用为「已验证」
- 发现矛盾叙述时须 `[!CONFLICT]` 并指向可核验源
- 压缩时待核实/观点条目须重新评估，不得默认升级

---

## 组件十一：角色切换纪律（复杂编排专用）

当主编排者需要在「复杂编排层」（如军团 L3）与「基础执行层」（如蜂群 L2 实测）之间切换时：

| 阶段 | 身份 | L1 持有 |
|------|------|---------|
| 进入执行层前 | 切换为执行身份 | 验收清单 + 代码路径 |
| 执行期间 | 执行身份 | 命令+输出+结果（禁发新编排子任务） |
| 退出回编排层 | 恢复编排身份 | 编排态 + 实测证据写入 §B |

> 诚实说明：多数 AI 平台无进程级隔离，此为主编排者的心智纪律。
