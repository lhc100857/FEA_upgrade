# 集成指南：如何将本协议接入你的治理体系

> 本指南帮助你（人类或 AI）将「主帅上下文生命周期管理协议 v2」集成到**任何**已有的多代理治理体系中。
> 不依赖特定文件路径、特定平台或特定命名约定。

---

## 第 1 步：放置新文件

### 1.1 协议技能文件

将 `PROTOCOL.md` 放入你的技能/知识库目录。

**适配要点**：
- 如果你的技能文件需要 frontmatter（如 YAML `name:` / `description:`），按你的规范加上
- 如果你的技能有编号体系，分配一个编号
- 文件内容无需修改——它是平台无关的

### 1.2 CHAIN-STATE 模板

将 `CHAIN-STATE-TEMPLATE.md` 放入你的持久化/落盘目录。

**适配要点**：
- 修改 §E 中的路径占位符为你的实际路径约定
- 如果你没有复杂编排（军团），§C/§E/§F 保持空即可

---

## 第 2 步：在治理规则中注入协议锚点

你需要在治理规则的**6 个语义位置**注入新条款。以下用**语义描述**定位，而非具体行号。

### 注入点 ①：规则文件触发词/描述区

**语义位置**：规则文件的元数据或触发关键词列表

**追加内容**（适配你的格式）：
> 上下文生命周期管理、三层模型、CHAIN-STATE、delta 汇流、会话切割

---

### 注入点 ②：冻结事实/形状保持相关条款

**语义位置**：描述「已拍板的事实不得被静默覆盖」的规则段

**插入内容**（适配你的措辞）：

> - **冻结条四态生命周期**：active（当前有效）→ superseded（被新条替代，须指向替代条 ID）→ archived（压缩时仅留 1 行墓碑；墓碑 >20 行时清理到最近 5 行）→ invalidated（被破坏性变更推翻，关联阶段门裁决须重验）。
> - **条件追踪硬上限**：同时 open 的条件性通过 ≤8 条；第 9 条到来时最旧 → expired 并记入风险登记。

---

### 注入点 ③：主编排者每轮状态汇报/续聊条款

**语义位置**：描述「每轮回复末尾保留当前任务状态」的规则段

**替换或追加**（核心变更——从全量重述改为增量）：

> 优先使用增量 delta 格式（`[+]` 新增 / `[~]` 变更 / `[=]` 验收进展 / `[!]` 待决 / `[!CONFLICT]` 矛盾 / `[→]` 下轮指令）。不可省段：目标句首行回显、active P0 变更、验收裁决变更。未变化的冻结条正文、已通过的验收详情可省。若已启用 CHAIN-STATE，delta 同步 patch 到 §G 区。

---

### 注入点 ④：跨轮记忆/持久化相关条款

**语义位置**：描述「子代理没有隐式记忆、需要主编排者传递信息」或「跨轮/跨会话信息保持」的规则段

**插入内容**：

> - **CHAIN-STATE 锚点**：每条任务链建议维护一份 CHAIN-STATE 文件作为工作记忆锚点。前 20 行（§A 精简下限区）须零读外文件即可回答目标句 + P0 + 裁决。
> - **会话切割与接力**：主编排者窗口达到 ≥15 轮（复杂编排）/ ≥20 轮（基础）或 ≥50k token 时，强烈建议切割会话。同一 P0 连续 3 轮失败时强制切割。切割时须写 HANDOFF 文件（全量 active 冻结条 + P0 + 最近 2 轮 delta + 下轮指令）。新会话启动优先级：HANDOFF > CHAIN-STATE > 从头。

---

### 注入点 ⑤：子代理使用/调度相关条款

**语义位置**：描述「如何分发任务给子代理、子代理如何返回结果」的规则段

**追加内容**：

> - **子代理回流格式约束**（复杂编排）：子任务角色须在任务尾部嵌入回流约束（≤800 字、结论 + 交付物指针 + 下游依赖 + 风险/待决）。子任务完成后主编排者按归档协议将结论写入 CHAIN-STATE §B、摘要写入 §F。

---

### 注入点 ⑥：规范栈/技能引用列表

**语义位置**：列出所有配套技能/文档引用的规则段

**追加**：协议技能文件的引用（路径用你的实际路径）。

---

### 注入点 ⑦：身份与意图三字段（新增条款或补强已有条款）

**语义位置**：如果你的体系已有「用户意图识别」「身份确认」「人设约束」类条款，在其附近补强。如果没有，在「冻结事实/形状保持」段后新增一节。

**插入内容**（适配你的措辞）：

> **身份与意图三字段**（每条任务链至少冻结一次；用户更正须重冻结）：
> 1. **执行者身份**：谁在请求（真实 / 假定）
> 2. **期望 AI 人设**：AI 应采用的专业边界与口吻
> 3. **产出下游用途**：产出物主要消费场景
>
> - 用户明示 → 标 A；AI 推断 → 标 B（待核实）；禁止 C 级冒充
> - 高风险动作/写码前用途模糊 → 必须追问
> - 人设要求冒充持证专业人士 → 拒绝或降级
> - 用途变更 = 契约变更 → 验收表须重冻结
> - 三字段属链上形状，适用形状保持纪律，不得静默覆盖
>
> 可选口令：用户说「身份开场」「先三问」等，AI 须先输出三个快速选择问题再承接需求。

**同步更新**：
- 如果你的 CHAIN-STATE 模板 §A 区尚未包含三字段，加一行 `- **身份三字段**: {执行者} | {人设} | {用途}`
- 如果你的汇流/续聊模板有状态字段列表，确保包含三字段及其 A/B/C 等级

---

### 注入点 ⑧：平台压缩联动与 wave-end 建议触发

**语义位置**：会话切割/续聊/状态持久化相关条款附近。如果你的体系有「wave」「阶段门（Gate）」「Phase」等多阶段概念，在阶段切换相关段落附近补强。

**插入内容**（适配你的措辞）：

> - **平台压缩联动（用户确认制）**：平台支持上下文压缩时，须遵守 **写盘→询问用户→用户确认后压缩→校验** 四步纪律。**压缩决策权归用户**——主编排者完成写盘后须向用户输出压缩建议并等待确认，**不得**自行触发或假设用户会执行。压缩后须 Read §A 校验目标句、P0、裁决、三字段未丢；不匹配则从 §B+§G 恢复。用户选择暂不压缩时，主编排者正常继续，不得催促。
> - **复杂编排 wave-end 建议触发**（如适用）：每个 wave 的阶段门裁决完成后，是建议上下文压缩的**最佳时机**。主编排者须先完成写盘（§C/§D/§F/§B/§A/§E 均已更新），然后**询问用户**是否执行压缩。
> - **压缩 vs 切割选择**：任务进展正常的膨胀用压缩（轻量）；深度失忆/跨链/连续失败用会话切割（重量）。

## 第 4 步：验证

1. 检查规则文件内部无逻辑矛盾（新条款不应与已有阶段门、验收纪律冲突）
2. 检查所有路径引用与实际文件位置一致
3. 如果有 linter/健康检查脚本，运行一次
4. 新条款均为**扩展**（新增冻结条生命周期、新增 delta 格式、新增 CHAIN-STATE），不删除/覆盖已有规则

## 常见适配问题

**Q: 我的规则文件不是 .mdc 格式**
A: 无所谓。本协议的注入点是**语义位置**，不是文件格式。你的规则可以是 Markdown、YAML、JSON、纯文本 system prompt——找到语义对应的段落插入即可。

**Q: 我没有「军团/复杂编排」**
A: 只用基础模式。CHAIN-STATE 中 §C/§E/§F 留空。组件六（常驻内容分级）和组件十（角色切换）跳过。

**Q: 我的子代理实现形态和示例不同**
A: 「子代理」可以是任何被主编排者调度的独立执行单元——IDE 内置的 Task/Sub-agent、对话式 AI 的 Artifacts、自动化框架的 sub-task/Agent/node 等。回流格式约束嵌入它们的 system prompt 或任务描述即可。

**Q: 我没有持久化目录**
A: 创建一个。最简可以是项目根下的 `.agent-state/` 或 `agent-memory/`。CHAIN-STATE 和 HANDOFF 放在里面。

**Q: 我在 Claude.ai 网页版/API 中用，没有文件系统**
A: 用 Artifacts 或对话内 code block 模拟 CHAIN-STATE。每轮手动 patch §G。HANDOFF 写成对话的最后一条消息，新对话粘贴到首条。效果次于文件但仍有效。

**Q: 预算数值不适合我的模型**
A: 按你的模型调整。原则：L1 不超过有效窗口的 40%，L2 不超过 10%，规则税用实际测量。
